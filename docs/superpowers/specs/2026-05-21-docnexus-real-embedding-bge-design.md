# DocNexus Real Embedding (bge-small-zh-v1.5) Design

Date: 2026-05-21

## Context

DocNexus has completed LadybugDB-based graph and vector recall integration on `main`.  
Current indexing and recall pipelines are stable, but embedding generation still uses local hash vectors, which limits semantic retrieval quality.

The next development target is to replace hash embedding with a real local embedding model while keeping existing MCP/CLI contracts and workflow semantics unchanged.

## Goal

Integrate real local embeddings with default model `BAAI/bge-small-zh-v1.5` (Node in-process inference), and make existing index/recall flow run end-to-end without changing user-facing commands.

## Scope

This iteration includes:

- Replace default embedding implementation with real local model embedding.
- Keep the same embedder call contract used by index and recall flows.
- Keep MCP tools and CLI command interface unchanged.
- Ensure LadybugDB write and recall continue to work with real vectors.

This iteration does not include:

- Model switching mechanism.
- Multi-model coexistence indexing.
- Auto migration or auto rebuild workflow.
- MCP-side final answer generation.
- Recall ranking strategy upgrade beyond embedding source replacement.

## Architecture And Change Boundary

1. Keep archive/index/recall lifecycle unchanged:
   - `archive_record` behavior stays the same.
   - `index upsert/delete` command semantics stay the same.
   - LadybugDB graph and vector storage schema stays the same.
   - MCP returns recall context only; no internal LLM answer generation.

2. Replace embedder implementation only:
   - Add a real embedder implementation module (for example `src/embedder-real.ts`) using local Node in-process inference.
   - Keep input/output contract compatible with current embedder usage (text in, fixed-dimension vector out).
   - Update embedder export wiring so index and recall paths use real embeddings by default.

3. Minimize touch surface:
   - `src/file-index.ts` and `src/recall.ts` only receive minimal dependency wiring changes if needed.
   - No business behavior refactor in this phase.

## Data Compatibility And Migration Policy

1. Embedding dimension consistency:
   - Effective vector dimension must match `EMBEDDING_DIMENSION` configuration and LadybugDB index expectations.
   - Runtime must fail fast on dimension mismatch instead of writing invalid vectors.

2. Existing indexed data:
   - Historical vectors generated by hash embedding remain readable but are considered lower quality.
   - After rollout, recommended operation is manual re-index/rebuild to refresh vectors with the real model.

3. No automatic migration in this phase:
   - No background migration jobs.
   - No automatic mixed-index reconciliation logic.
   - Use existing manual commands (`index upsert` / `index delete`) for controlled replacement.

## Verification Plan

1. Unit validation:
   - Real embedder returns stable fixed-dimension vectors.
   - For the same input under same model/runtime, output remains deterministic enough for index/recall consistency checks.

2. Integration validation:
   - `index upsert` persists vectors into LadybugDB successfully.
   - `recall` returns chunk context plus graph-related fields as expected.

3. Regression gates:
   - `npm test`
   - `npm run typecheck`
   - `npm run build`

4. CLI smoke check:
   - Run a Chinese query through `docnexus recall "<query>"`.
   - Confirm result shape still contains matched chunk and concept relationship context fields.

## Risks And Mitigations

- Model runtime/dependency risk:
  - Mitigation: keep replacement scope narrow to embedder module and integration points only.
- Dimension mismatch risk:
  - Mitigation: explicit runtime assertion against configured dimension.
- Mixed old/new vector quality variance:
  - Mitigation: document manual rebuild recommendation as part of rollout.

## Deliverables

- Specified real embedding default wired into current index/recall pipeline.
- Updated tests to cover embedder and recall path expectations with real vectors.
- Updated docs reflecting that hash embedding is replaced by local real embedding (default `bge-small-zh-v1.5`).
