# DocNexus LadybugDB Graph Recall Design

Date: 2026-05-20

## Context

DocNexus currently has a working local memory loop:

- Skills refine source content into Markdown and metadata.
- MCP archives source, document, and metadata in `.docnexus/records/<id>/`.
- CLI/MCP explicitly index files through `upsert_file_index`.
- SQLite stores file lifecycle, chunks, embeddings, and index audit events.
- Recall loads SQLite chunks and ranks them in TypeScript with `LocalHashEmbedder`.

The original `docPlan.md` calls for an embedded graph/vector store with LadybugDB, HNSW vector search, document/chunk/concept nodes, and graph traversal around matched chunks.

The next phase will connect LadybugDB first, while preserving the existing command-driven workflow.

## Goal

Add LadybugDB as the project-local graph and vector recall store.

This phase should store chunk embeddings and metadata-derived graph relationships in `.docnexus/store.lbug`, then use LadybugDB vector search and graph traversal for `recall`.

SQLite remains the archive ledger and file lifecycle/audit store. LadybugDB becomes the recall substrate.

## Chosen Approach

Use LadybugDB for graph and vector storage, keep SQLite for records and index lifecycle.

This is option A:

- `.docnexus/index.sqlite` keeps `records`, `indexed_files`, `file_chunks`, and `index_events`.
- `.docnexus/store.lbug` stores `Project`, `Document`, `Chunk`, `Concept`, and graph relationships.
- `upsert_file_index` writes to both SQLite and LadybugDB.
- `delete_file_index` deletes recall graph/vector data from LadybugDB and marks the SQLite row as deleted.
- `recall` uses LadybugDB vector index first, then expands graph context.

The embedding generator remains the existing `Embedder` interface. The first LadybugDB implementation can still use `LocalHashEmbedder`; the storage and query path should not depend on that concrete class. This keeps the graph/vector work unblocked and leaves a narrow replacement point for the project's later lightweight embedding model.

## Non-Goals

This phase will not add:

- automatic file watching
- automatic capture
- external model provider integration
- MCP-side final answer generation
- HTTP server bridge
- global project registry
- destructive full rebuild/clean commands

## Storage Layout

DocNexus keeps all data project-local:

```text
<project-root>/
└── .docnexus/
    ├── index.sqlite
    ├── store.lbug
    ├── records/
    │   └── <record_id>/
    │       ├── source.md
    │       ├── document.md
    │       └── metadata.json
    └── schemas/
        └── metadata.schema.json
```

`index.sqlite` remains the source of truth for file lifecycle and audit history. `store.lbug` is rebuildable recall state derived from indexed file content and archived metadata.

## LadybugDB Schema

Create `src/ladybug-store.ts` to own schema initialization and graph/vector operations.

The schema should use stable string IDs already generated by DocNexus.

```cypher
CREATE NODE TABLE Project(
  id STRING,
  name STRING,
  root_path STRING,
  PRIMARY KEY (id)
);

CREATE NODE TABLE Document(
  id STRING,
  file_id STRING,
  record_id STRING,
  title STRING,
  path STRING,
  summary STRING,
  content_hash STRING,
  updated_at STRING,
  PRIMARY KEY (id)
);

CREATE NODE TABLE Chunk(
  id STRING,
  file_id STRING,
  document_id STRING,
  text STRING,
  text_hash STRING,
  chunk_index INT64,
  embedding FLOAT[64],
  PRIMARY KEY (id)
);

CREATE NODE TABLE Concept(
  id STRING,
  name STRING,
  type STRING,
  description STRING,
  PRIMARY KEY (id)
);

CREATE REL TABLE HAS_DOCUMENT(FROM Project TO Document);
CREATE REL TABLE HAS_CHUNK(FROM Document TO Chunk);
CREATE REL TABLE NEXT_CHUNK(FROM Chunk TO Chunk);
CREATE REL TABLE MENTIONS(FROM Document TO Concept);
CREATE REL TABLE DEPENDS_ON(FROM Concept TO Concept);
CREATE REL TABLE RELATES_TO(FROM Concept TO Concept);
CREATE REL TABLE IMPLEMENTS(FROM Concept TO Concept);
CREATE REL TABLE REPLACES(FROM Concept TO Concept);
CREATE REL TABLE DECIDES(FROM Concept TO Concept);
```

Use `FLOAT[64]` while `LocalHashEmbedder` remains the default 64-dimensional embedder. When the real lightweight embedding model lands, this dimension must become a single constant shared by the embedder and Ladybug schema.

After schema creation, load the vector extension and create the HNSW index:

```cypher
INSTALL vector;
LOAD vector;

CALL CREATE_VECTOR_INDEX(
  'Chunk',
  'chunk_vector_index',
  'embedding',
  metric := 'cosine'
);
```

Schema initialization must be idempotent. If LadybugDB does not support `IF NOT EXISTS` for a statement, the wrapper should catch "already exists" errors and continue only for those known idempotency cases.

## Entity and Relationship Mapping

DocNexus metadata already includes:

- `entities[].name`
- `entities[].type`
- `entities[].description`
- `relationships[].from`
- `relationships[].to`
- `relationships[].type`
- `relationships[].description`

When an indexed file is linked to a `record_id`, `upsert_file_index` should read `.docnexus/records/<record_id>/metadata.json` and map metadata into LadybugDB:

- each metadata entity becomes a `Concept`
- the indexed document gets `MENTIONS` edges to those concepts
- each metadata relationship becomes one typed concept-to-concept edge

Relationship type mapping:

```text
depends_on -> DEPENDS_ON
mentions   -> RELATES_TO
relates_to -> RELATES_TO
implements -> IMPLEMENTS
replaces   -> REPLACES
decides    -> DECIDES
```

Concept IDs should be deterministic from normalized entity name and type, for example `concept_<sha256(type:name)>`. This avoids duplicate concept nodes across documents.

If a relationship references an entity that does not exist in `entities`, create a minimal `Concept` with:

- `name` from the relationship endpoint
- `type = "other"`
- `description = "Referenced by metadata relationship."`

## Upsert Flow

`upsert_file_index` remains the only add/update command.

Flow:

1. Resolve and validate the user-specified file path.
2. Read content and compute `content_hash`.
3. Use SQLite to decide `created`, `updated`, or `noop`.
4. For `noop`, return without rewriting LadybugDB.
5. For `created` or `updated`, chunk the content and generate embeddings.
6. Update SQLite lifecycle rows and chunks as the existing implementation does.
7. Replace the document's LadybugDB recall graph:
   - delete old `Document` and attached `Chunk` nodes for the file
   - keep shared `Concept` nodes
   - create or update `Project`
   - create `Document`
   - create ordered `Chunk` nodes
   - create `HAS_DOCUMENT`, `HAS_CHUNK`, and `NEXT_CHUNK`
   - if `record_id` is present, map metadata entities and relationships
8. Write the existing SQLite `index_events` row.

SQLite should remain the source of truth for whether the file is currently indexed. LadybugDB should be treated as derived state.

## Delete Flow

`delete_file_index` remains explicit by `file_path` or `file_id`.

Flow:

1. Resolve the indexed file from SQLite.
2. Delete that file's chunks from SQLite.
3. Mark `indexed_files.index_state = "deleted"`.
4. Delete the corresponding LadybugDB `Document`, attached `Chunk` nodes, and document-local edges.
5. Keep shared `Concept` nodes unless they become cleanup targets in a later rebuild command.
6. Write a SQLite `index_events` row.

Deletion affects recall only. It must not delete archived source, document, or metadata.

## Recall Flow

`recall` should return context chunks and graph context, not a final answer.

Flow:

1. Validate `query`.
2. Generate query embedding with the configured `Embedder`.
3. Query LadybugDB `chunk_vector_index`.
4. For each matched chunk, return:
   - document title/path/summary
   - matched chunk
   - next chunk when available
   - directly mentioned concepts
   - one-hop related concepts
   - similarity score
5. Sort by score according to LadybugDB vector search semantics.
6. Enforce the existing `limit` defaults and maximum.

Target output:

```json
{
  "query": "string",
  "results": [
    {
      "file_id": "string",
      "file_path": "string",
      "record_id": "optional string",
      "document_title": "string",
      "document_summary": "string",
      "chunk_id": "string",
      "chunk_index": 0,
      "score": 0.92,
      "text": "string",
      "next_text": "optional string",
      "concepts": ["DocNexus", "MCP"],
      "related_concepts": ["LadybugDB"]
    }
  ]
}
```

If `.docnexus/store.lbug` has no chunks, return an empty result list. Do not silently fall back to SQLite recall in the first implementation; fallback behavior would hide LadybugDB integration failures.

## CLI and MCP Contracts

Existing public tool and CLI names stay unchanged:

- `upsert_file_index`
- `delete_file_index`
- `recall`
- `index_status`
- `docnexus index upsert <file_path> --name <file_name> --record-id <record_id>`
- `docnexus index delete --file <file_path>`
- `docnexus index delete --id <file_id>`
- `docnexus recall "<query>" --limit 5`
- `docnexus index status`

The response shape for `recall` gains graph context fields. Existing callers that only read `results[].text` remain compatible.

## Error Handling

Errors should remain short and actionable:

- `LadybugDB dependency is not installed`
- `failed to initialize LadybugDB store`
- `failed to create vector index`
- `embedding dimension mismatch`
- `failed to write LadybugDB graph`
- `failed to query LadybugDB recall`

If SQLite update succeeds but LadybugDB write fails, the operation should return an error and write a failed `index_events` row. A later retry of `upsert_file_index` should be able to repair derived LadybugDB state.

## Testing Strategy

Add focused tests around the new LadybugDB layer. If LadybugDB cannot run in the current Node test environment, wrap integration tests with a runtime availability check and keep pure mapping tests always enabled.

Unit tests:

- concept ID generation is deterministic
- metadata relationship types map to expected graph edge labels
- embedding dimension constant matches schema dimension

Integration tests:

- initializes `.docnexus/store.lbug`
- upsert creates document and chunk nodes
- upsert with `record_id` creates concepts and relationships
- recall returns vector-ranked chunks with graph context
- delete removes document/chunks from recall

Verification:

```bash
npm test
npm run typecheck
npm run build
```

## Success Criteria

This phase is complete when:

1. `docnexus index upsert <file> --record-id <id>` writes chunk embeddings to LadybugDB.
2. Metadata entities and relationships are stored as LadybugDB concepts and edges.
3. `docnexus recall "<query>"` queries LadybugDB vector index and returns graph context.
4. `docnexus index delete` removes the file from LadybugDB-backed recall.
5. Existing archive, validation, listing, and status behavior remains intact.
